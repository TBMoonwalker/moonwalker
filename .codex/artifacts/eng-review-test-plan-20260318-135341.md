# Test Plan

Generated by `plan-eng-review`

- Date: `2026-03-18`
- Branch: `feature/optimizing-stabilizing`
- Goal: `Refactor the main code-smell hotspots without changing Moonwalker's single-node, single-instance operating model.`

## Affected Pages or Routes

- `/config` - verify config load/save behavior stays stable while the frontend form logic and backend config contract are cleaned up.
- `/config/all`, `/config/single/{key}`, `/config/multiple` - verify public config responses, save semantics, and validation after moving away from leaked `_cache` and implicit sentinel values.
- `/orders/buy/{symbol}/{ordersize}`, `/orders/sell/{symbol}`, `/orders/stop/{symbol}` or their planned POST replacements - verify manual trade actions remain correct and safe across multiple dashboard clients.
- `/trades/open`, `/trades/closed`, `/trades/unsellable`, `/statistic/profit` - verify WebSocket fan-out still works while read paths and runtime boundaries are simplified.

## Key Interactions To Verify

- Open `/config`, load persisted values, edit, save, and reload - expected result: values round-trip correctly with no silent clearing or type coercion regressions.
- Use desktop and mobile clients against the same instance - expected result: both receive the same trade/statistics updates and manual actions do not duplicate unexpectedly.
- Trigger manual buy, sell, and stop from the dashboard - expected result: state transitions are correct and reflected in the next streamed payload.
- Restart the app after config changes - expected result: watcher, signal, and statistics services boot with the same effective config as before the refactor.

## Edge Cases

- Clearing optional config fields - expected behavior: fields are explicitly cleared through the new contract, not via hidden truthiness rules.
- Invalid config payload types - expected behavior: clear 4xx validation errors, no partial mutation.
- Multiple dashboard clients clicking trade actions near-simultaneously - expected behavior: no unintended duplicate orders.
- Watcher/runtime internal errors - expected behavior: failures are logged clearly and the instance remains recoverable without hidden stuck state.

## Critical Paths

- Config lifecycle: load config -> edit in UI -> save -> runtime reload -> reopen UI -> values match.
- Manual trading lifecycle: dashboard action -> backend order handling -> trade persistence -> statistics/trades streams update.
- Single-instance runtime lifecycle: process starts -> long-lived services initialize -> watchers/signals run -> dashboard clients subscribe concurrently.
