# Test Plan

Generated by `plan-eng-review`

- Date: `2026-03-18`
- Branch: `feature/optimizing-stabilizing`
- Goal: `Validate the current Moonwalker project shape end-to-end, with emphasis on config management, trading actions, live dashboard streams, backup/restore, and operational safety.`

## Affected Pages or Routes

- `/` - verify the trading dashboard loads, renders statistics, and handles live WebSocket updates without stale or duplicate data.
- `/config` - verify configuration loads from persisted state, validates required fields, saves successfully, and preserves secrets/advanced settings correctly.
- `/orders/sell/{symbol}` - verify sell actions trigger exactly once and do not accidentally repeat due to refresh, retries, or browser prefetch behavior.
- `/orders/buy/{symbol}/{ordersize}` - verify manual add-funds flow creates the intended safety order only once and rejects invalid amounts.
- `/orders/stop/{symbol}` - verify stop flow removes the deal from bot control without corrupting trade history.
- `/config/backup/export` and `/config/backup/restore` - verify config-only and full backup flows, including restored history refresh behavior and failure reporting.
- `/trades/open`, `/trades/closed`, `/trades/unsellable`, `/statistic/profit` - verify fan-out streams recover cleanly from disconnects and do not leak stale payloads between reconnects.

## Key Interactions To Verify

- Open dashboard with a complete config on `/` - expected result: user stays on dashboard and live widgets populate.
- Open dashboard with incomplete config on `/` - expected result: router redirects to `/config?setup=required`.
- Save exchange and signal settings on `/config` - expected result: `/config/multiple` persists values, watcher/signal reload cleanly, and subsequent `/config/all` reflects the update.
- Trigger manual buy from the open-trades table on `/` - expected result: trade history and open-trade state update consistently, with no duplicate safety order.
- Trigger sell or stop from the open-trades table on `/` - expected result: only one action is executed even if the UI is clicked twice or the page refreshes.
- Download and restore a config-only backup on `/config` - expected result: config rows are replaced, runtime reloads, and existing trade data remains intact.
- Download and restore a full backup on `/config` - expected result: config plus trade tables are replaced, ticker history is rehydrated for active trades, and failures are surfaced clearly.
- Disconnect and reconnect WebSocket consumers on `/` - expected result: latest payload is replayed once and stream status recovers without requiring a full page reload.

## Edge Cases

- Missing or invalid Redis availability during config save - expected behavior: local save still succeeds, but any multi-process reload expectation is explicitly validated.
- Switching signal plugin to `csv_signal` while open trades exist - expected behavior: request is blocked with a clear error.
- Visiting the app from a non-local origin - expected behavior: confirm whether the deployment is intentionally open or protected by network/auth controls.
- Backup restore with malformed JSON or missing trade data - expected behavior: restore is rejected with a clear 4xx response and no partial data loss.
- Exchange/demo symbol missing after config change - expected behavior: watcher logs a warning, skips the symbol safely, and keeps the rest of the runtime healthy.
- Browser refresh or crawler/prefetch against GET order routes - expected behavior: no unintended order placement or stop/sell action occurs.

## Critical Paths

- First-run setup: load app -> detect incomplete config -> complete `/config` form -> save -> return to dashboard -> live stats and trade tabs load.
- Active trading loop: signal plugin enqueues symbols -> watcher streams market data -> DCA evaluates -> order executes -> trades/statistics streams update.
- Manual intervention loop: operator uses buy/sell/stop from dashboard -> backend mutates trade state -> UI reflects the result on next stream payload.
- Recovery loop: export backup -> wipe/restore config and trade state -> history refill runs -> dashboard becomes consistent again after reconnect.
